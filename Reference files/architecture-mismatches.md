## Audit Date: 2026-04-26

### Fixed in this session
* **Reader/ReaderViewModel — main-thread blocking when launching from Library:** `loadFromLibrary(fileURL:)` runs on `@MainActor` (the whole class is) and synchronously called `Data(contentsOf: fileURL)` and `data.base64EncodedString()`. For a 10–20 MB EPUB this freezes the UI for hundreds of milliseconds, violating the `SWIFT_CONC` invariant ("All Core Data writes on backgroundContext. All UI updates on MainActor" implies non-UI work must come off the main actor) and the master-plan `epub_open < 1.5s` perf target. The matching `FileImporter.importSingleEPUBForReader` path already hops to a detached task — only the library-launch path was wrong.
  * **Fix applied:** Extracted the I/O + base64 + JS-escape work into a new `nonisolated static func loadAndEncode(fileURL:)` that runs inside `Task.detached(priority: .userInitiated)` and returns `(encoded, escaped)`. `loadFromLibrary` now `await`s that helper before assigning the published properties on the main actor, mirroring the pattern already used in `FileImporter.importSingleEPUBForReader`.
* **Library/LibraryView — empty state shown on zero-result search:** `LibraryView.body` gated the empty state on `viewModel.filteredBooks.isEmpty`, so any search term that didn't match (e.g. typing "xyz" with 50 books in the library) replaced the grid/list with the "Import your first book" CTA. Master-plan task 1.6 is explicit: "if `books.isEmpty` show empty state illustration + 'Import your first book' text + import button. Else: switch on displayMode → LazyVGrid … or List. Filter by searchQuery against title+author." — i.e. a search miss should still render the (empty) filtered content view, not the import CTA.
  * **Fix applied:** Switched the conditional to `viewModel.books.isEmpty` so the empty state only appears for an actually-empty library; a search miss now falls through to the regular grid/list with no rows.

### Pending / Unfixed Mismatches
* **Reader/PageCurlViewController — three-slot pool drifts out of sync with the rendition:** Each pool slot owns its own `EPUBBridge` + `WKWebView` + epub.js `book` instance and `loadBook(...)` is broadcast to all three at start, but pool rotation only nudges adjacent slots one page (`prevPage()` / `nextPage()`) regardless of where the user actually is. After a few page turns or any chapter jump (via TOC, scrubber, or `displayCFI`), the recycled prev/next slots are no longer at the user's neighbouring pages, so the curl animation reveals stale or wrong content. Master-plan task 2a.5 specifies "the page VC manages current and adjacent view controllers each hosting one EPUBWebView from a pool of 3 … Swift controls which `rendition.display()` is called". The current implementation does not call `rendition.display(adjacentCFI)` on the pool slots when the current slot relocates.
  * **Why it wasn't fixed:** This is a design-level fix (slots need to mirror the current slot's CFI ± one page on every `relocated`, which requires knowing CFIs of adjacent pages from epub.js — a non-trivial JS round-trip via `book.locations` or `rendition.location` lookups) and would require extending the bridge with a "navigate to adjacent CFI" channel. Doing it half-way would leave the reader in a worse visible state than today.
  * **Recommended Next Steps:** Extend `EPUBBridge` / `reader.html` with `displayPrev(cfi)` / `displayNext(cfi)` helpers driven from epub.js's `Locations`, then on every `onRelocated` callback push the adjacent CFIs into `prevSlot` / `nextSlot` before the user can curl to them. Also consider whether the master-plan invariant "max 3 instances pooled" can be satisfied with a *single* shared `book` JS instance rendered into three different viewers (lower memory, simpler sync) instead of three independent `ePub()` parses.
* **Reader/PageCurlReaderView — `@State` written during view creation:** `PageCurlReaderView` invokes `onCreated(vc)` synchronously inside `makeUIViewController`, and the caller (`ReaderView`) assigns the result to a `@State var pageCurlVC`. SwiftUI prohibits modifying state during view-body evaluation; this currently "works" in practice but is the kind of pattern that emits "Modifying state during view update" runtime warnings and can break in future SDKs.
  * **Why it wasn't fixed:** The right fix is to expose the controller through the view-model (e.g. an `@MainActor` reference set inside the coordinator and read by `ReaderView` via the view-model) rather than plumbing it back through a closure. That's a small but invasive refactor to the freshly-landed task 2a.5 wiring, and not safe to do under a single-pass audit without exercising the page-curl flow on a device.
  * **Recommended Next Steps:** Move `pageCurlVC` ownership onto `ReaderViewModel` (or its coordinator) and have `PageCurlReaderView` write to it via the view-model instead of a SwiftUI `@State` binding.
